# db — Chainable Collection API over Managed SQLite

The `db` namespace provides a typed, chainable collection API for MindStudio's managed SQLite databases. It feels like working with JS collections — filter, sort, push, update, remove — but every operation is a database query. The `await` is the only signal that you're crossing the wire.

## Quick start

```ts
import { db, User } from '@mindstudio-ai/agent';

// Define a typed table (module scope — lazy, no HTTP)
interface Order {
  // System fields — required in the interface, auto-managed by platform
  id: string;
  createdAt: number;
  updatedAt: number;
  lastUpdatedBy: string;

  // Your fields
  item: string;
  amount: number;
  status: 'pending' | 'approved' | 'rejected';
  requestedBy: User;  // branded string type for user references
}

const Orders = db.defineTable<Order>('orders');

// If the app has multiple databases, specify which one:
// const Orders = db.defineTable<Order>('orders', { database: 'main' });
```

### Reading data

```ts
// Get by ID
const order = await Orders.get('abc-123');

// Filter with a predicate — compiles to SQL when possible
const active = await Orders.filter(o => o.status === 'approved').take(10);

// Chain operations (lazy — nothing executes until await)
const recent = await Orders
  .filter(o => o.status === 'approved')
  .sortBy(o => o.createdAt)
  .reverse()
  .take(20);

// Find one
const firstPending = await Orders.findOne(o => o.status === 'pending');

// Aggregations
const total = await Orders.count();
const pendingCount = await Orders.count(o => o.status === 'pending');
const hasAny = await Orders.some(o => o.status === 'pending');
const allDone = await Orders.every(o => o.status === 'approved');
const empty = await Orders.isEmpty();

// Min/max (returns the full row, not just the value)
const cheapest = await Orders.min(o => o.amount);
const biggest = await Orders.max(o => o.amount);

// Grouping
const byStatus = await Orders.groupBy(o => o.status);
// Map { 'pending' => [...], 'approved' => [...] }

// Pagination
const page2 = await Orders
  .sortBy(o => o.createdAt)
  .reverse()
  .skip(50)
  .take(50);
```

### Writing data

```ts
// Insert — system fields (id, createdAt, etc.) are auto-populated
const order = await Orders.push({
  item: 'Laptop',
  amount: 999,
  status: 'pending',
  requestedBy: auth.userId,
});
console.log(order.id);        // UUID generated by platform
console.log(order.createdAt); // unix timestamp (ms)

// Insert multiple
const orders = await Orders.push([
  { item: 'Monitor', amount: 300, status: 'pending', requestedBy: auth.userId },
  { item: 'Keyboard', amount: 50, status: 'pending', requestedBy: auth.userId },
]);

// Partial update — only provided fields change
const updated = await Orders.update(order.id, { status: 'approved' });

// Delete by ID
await Orders.remove(order.id);

// Delete matching rows — returns count
const removed = await Orders.removeAll(o => o.status === 'rejected');

// Delete everything
await Orders.clear();
```

### Using with auth

```ts
import { db, auth, Roles } from '@mindstudio-ai/agent';

const Orders = db.defineTable<Order>('orders');

export const getDashboard = async () => {
  const userId = auth.userId;

  // Everyone sees their own orders
  const myOrders = await Orders
    .filter(o => o.requestedBy === userId && o.status !== 'inactive')
    .sortBy(o => o.createdAt)
    .reverse()
    .take(25);

  // Admins see org-wide stats
  let totalCount;
  if (auth.hasRole(Roles.admin)) {
    totalCount = await Orders.count();
  }

  return { myOrders, totalCount };
};
```

### Using time helpers

```ts
import { db } from '@mindstudio-ai/agent';

// Timestamps are unix ms — use helpers for readable time math
const stale = await Orders.filter(
  o => o.status === 'pending' && o.createdAt < db.ago(db.days(2))
);

// Update with current timestamp
await Orders.update(order.id, { approvedAt: db.now() });

// Composable durations
const cutoff = db.ago(db.days(7) + db.hours(12));
```

### Resolving users

```ts
import { resolveUser } from '@mindstudio-ai/agent';

// When you need display info for a User field
const user = await resolveUser(order.requestedBy);
if (user) {
  console.log(user.name, user.email, user.profilePictureUrl);
}
```

## How it works

```
User code                          SDK internals                    Platform
─────────                          ─────────────                    ────────
Orders.filter(pred).take(10)
  │
  ├─ Query accumulates operations  (lazy, nothing executes)
  │
  └─ await ──────────────────────► compilePredicate(pred)
                                    │
                                    ├─ Success: SQL WHERE clause
                                    │  buildSelect('orders', { where, limit: 10 })
                                    │  ──► executeStep('queryAppDatabase', { sql })
                                    │                                     ──► SQLite
                                    │
                                    └─ Failure: JS fallback
                                       buildSelect('orders', {})  ← fetch ALL rows
                                       rows.filter(pred).slice(0, 10)
                                       ⚠ warning logged to stderr
```

### SQL fast path vs JS fallback

When you write a filter predicate, the SDK attempts to compile it to a SQL WHERE clause:

**Compiles to SQL** (efficient, single query):
```ts
.filter(o => o.status === 'active')           // WHERE status = 'active'
.filter(o => o.amount > 5000)                 // WHERE amount > 5000
.filter(o => o.a === 1 && o.b === 2)          // WHERE a = 1 AND b = 2
.filter(o => o.a === 1 || o.b === 2)          // WHERE a = 1 OR b = 2
.filter(o => ['a','b'].includes(o.status))    // WHERE status IN ('a', 'b')
.filter(o => o.name.includes('test'))         // WHERE name LIKE '%test%'
.filter(o => o.field != null)                 // WHERE field IS NOT NULL
.filter(o => o.field === null)                // WHERE field IS NULL
.filter(o => !o.deleted)                      // WHERE deleted = 0
```

**Falls back to JS** (fetches all rows, filters in memory):
```ts
.filter(o => o.name.startsWith('A'))          // function call — not compilable
.filter(o => /^PO-\d+/.test(o.poNumber))      // regex
.filter(o => o.a + o.b > 100)                 // computed expression
.filter(o => someSet.has(o.id))               // complex closure
```

Both paths produce **identical results**. The fallback logs a warning so you can optimize if needed:
```
[mindstudio] Filter on orders could not be compiled to SQL — scanning 342 rows in JS
```

For most apps (hundreds to low thousands of rows), the fallback is fast enough.

## Table API reference

### Reads — direct (return Promises)

| Method | Description | SQL |
|--------|-------------|-----|
| `get(id)` | Single row by ID, or null | `SELECT * WHERE id = ?` |
| `findOne(pred)` | First matching row, or null | `SELECT * WHERE ... LIMIT 1` |
| `count(pred?)` | Count rows (all, or matching predicate) | `SELECT COUNT(*)` |
| `some(pred)` | True if any row matches | `SELECT EXISTS(...)` |
| `every(pred)` | True if all rows match | `SELECT NOT EXISTS(... WHERE NOT ...)` |
| `isEmpty()` | True if table has zero rows | `SELECT NOT EXISTS(SELECT 1 ...)` |
| `min(fn)` | Row with minimum value for field | `ORDER BY field ASC LIMIT 1` |
| `max(fn)` | Row with maximum value for field | `ORDER BY field DESC LIMIT 1` |
| `groupBy(fn)` | Group rows into `Map<K, T[]>` | Fetch all, group in JS |

### Reads — chainable (return `Query<T>`)

| Method | Description |
|--------|-------------|
| `filter(pred)` | Add a WHERE condition |
| `sortBy(fn)` | Set ORDER BY field (ascending) |

### Query chain methods

Chain methods return a new immutable Query. Nothing executes until you `await`.

| Method | Description |
|--------|-------------|
| `.filter(pred)` | Add another WHERE condition (ANDed with previous filters) |
| `.sortBy(fn)` | Set ORDER BY field |
| `.reverse()` | Flip sort direction (ASC ↔ DESC) |
| `.take(n)` | Limit results (SQL LIMIT) |
| `.skip(n)` | Skip first n results (SQL OFFSET) |

### Query terminal methods (execute the query)

| Method | Returns | Description |
|--------|---------|-------------|
| `await query` | `T[]` | Execute and return all matching rows |
| `.first()` | `T \| null` | First matching row |
| `.last()` | `T \| null` | Last matching row (flips sort direction) |
| `.count()` | `number` | Count of matching rows |
| `.some()` | `boolean` | True if any row matches |
| `.every()` | `boolean` | True if all rows match |
| `.min(fn)` | `T \| null` | Row with minimum value |
| `.max(fn)` | `T \| null` | Row with maximum value |
| `.groupBy(fn)` | `Map<K, T[]>` | Group results by field |

### Writes

| Method | Returns | Description |
|--------|---------|-------------|
| `push(data)` | `T` | Insert one row, returns created row with system fields |
| `push(data[])` | `T[]` | Insert multiple rows |
| `update(id, partial)` | `T` | Partial update by ID, returns updated row |
| `remove(id)` | `void` | Delete row by ID |
| `removeAll(pred)` | `number` | Delete matching rows, returns count removed |
| `clear()` | `void` | Delete all rows |

## System columns

Every row has platform-managed columns. They are **included in read results** but **excluded from write inputs** (`push()` and `update()` strip them automatically):

| Column | Type | Description |
|--------|------|-------------|
| `id` | `string` | UUID primary key, generated on INSERT |
| `createdAt` | `number` | Unix timestamp (ms), set on INSERT |
| `updatedAt` | `number` | Unix timestamp (ms), set on INSERT and every UPDATE |
| `lastUpdatedBy` | `string` | Run ID that last wrote this row |

Your table interface must include these fields, but you never set them:

```ts
interface Order {
  // System fields — include in interface, never in push/update
  id: string;
  createdAt: number;
  updatedAt: number;
  lastUpdatedBy: string;

  // Your fields — these are what you pass to push() and update()
  item: string;
  amount: number;
  status: 'pending' | 'approved' | 'rejected';
}
```

## Time helpers

All timestamps are unix milliseconds. The `db` namespace includes helpers for readable time math:

| Helper | Returns | Example |
|--------|---------|---------|
| `db.now()` | `number` | Current unix timestamp (ms) |
| `db.days(n)` | `number` | Duration in ms (`n * 86400000`) |
| `db.hours(n)` | `number` | Duration in ms (`n * 3600000`) |
| `db.minutes(n)` | `number` | Duration in ms (`n * 60000`) |
| `db.ago(ms)` | `number` | Unix timestamp for `now - ms` |
| `db.fromNow(ms)` | `number` | Unix timestamp for `now + ms` |

Durations are composable with `+`:
```ts
const cutoff = db.ago(db.days(7) + db.hours(12));  // 7.5 days ago
```

## User type

The `User` type is a branded string for platform user references. Columns with schema type `'user'` store values with a `@@user@@` prefix in SQLite. The SDK handles this transparently:

- **On read**: `@@user@@550e8400-...` → `550e8400-...`
- **On write**: `550e8400-...` → `@@user@@550e8400-...`

Application code always works with clean UUID strings. Use `resolveUser()` when you need display info.

## Database resolution

`defineTable()` needs to know which database contains the table:

1. **Single database** (common case) — used automatically. No need to specify.
2. **Explicit database** — pass `{ database: 'name-or-id' }` as the second argument.
3. **Multiple databases, no hint** — the SDK searches all databases by table name. Works if table names are unique.

```ts
// Single database — just works
const Orders = db.defineTable<Order>('orders');

// Multiple databases — specify which one
const Orders = db.defineTable<Order>('orders', { database: 'main' });
const Logs = db.defineTable<LogEntry>('entries', { database: 'analytics' });
```

## Context hydration

`db.defineTable()` is lazy — no HTTP is triggered. Context (database metadata, column schemas) is fetched automatically on the first query execution. Inside the MindStudio sandbox, context is resolved from the CALLBACK_TOKEN or pre-populated globals.

## Files

| File | Purpose |
|------|---------|
| `index.ts` | `createDb()` factory, `Db` interface, time helpers, table name → database resolution |
| `table.ts` | `Table<T>` class — the full read/write collection API |
| `query.ts` | `Query<T>` class — lazy chainable builder, SQL/JS dual execution |
| `predicate.ts` | Predicate compiler — tokenizer + recursive descent → SQL WHERE |
| `sql.ts` | SQL string builders, value escaping, row deserialization |
| `types.ts` | Internal types (Predicate, Accessor, TableConfig, SystemFields, etc.) |
